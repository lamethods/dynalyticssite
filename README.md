# Dynasite

A single, self-contained, **searchable hub** for the Saqr / López-Pernas R &
network-analysis ecosystem: every package, documentation site, vignette,
tutorial, and `lamethods.org` book chapter in one place — with **every link
verified** at build time and weekly via CI.

Open `index.html` (served over http) and start typing. No backend, no build
step to *view* it; the page reads a generated `catalog.json` and searches it
client-side with [Fuse.js](https://fusejs.io) (vendored locally, works offline).

## What's indexed

`catalog.json` (regenerated by `build/harvest.R`) currently holds **94 entries**:

| Type | Count | Source |
|------|-------|--------|
| Packages | 17 | local `*/DESCRIPTION` across the workspace |
| Vignettes / articles | 19 | each package's `vignettes/*.Rmd` → its live pkgdown `/articles/` |
| Tutorials | 10 | curated `sonsoles.me/posts/*` |
| Book chapters | 44 | live `lamethods.github.io/book{1,2}/sitemap.xml` |
| Sites / book home / extras | 4 | curated |

CRAN packages carry a green `CRAN <version>` badge; a coloured status dot shows
each entry's last verification result (green = verified, amber = redirect,
grey = local/unpublished).

## Layout

```
index.html              # the whole UI (loads assets/catalog.js, d3, chord, app)
catalog.json            # GENERATED — do not hand-edit; rerun the build
assets/
  app.js  style.css  chord.js  d3.min.js  catalog.js   # catalog.js is GENERATED
build/
  sources.json          # curated seed: doc-site URLs, CRAN flags, owners, tags,
                        #   posts, lamethods volumes + chapter titles, about, papers
  harvest.mjs           # DESCRIPTIONs + sources.json + sitemaps -> catalog.json (+ catalog.js)
  verify_links.mjs      # checks every http URL (browser UA), writes status, fails on broken
papers/                 # bundled PDFs (e.g. dynalytics-2026.pdf)
CNAME                   # custom domain for GitHub Pages: dynalytics.lamethods.org
.github/workflows/
  linkcheck.yml         # weekly cron re-verification; opens a 'linkrot' issue on breakage
```

`build/sources.json` is the single curation layer. To add a package, add a row
there (and ensure its `DESCRIPTION` is reachable under `workspace_root`); to
correct a doc-site URL, edit its `docs` field. Per-package escape hatches:
`"no_github": true` (repo not public), `"exclude_articles": ["slug"]` (vignette
not published to the live site).

## Build & verify

The build is **Node** (no dependencies — uses built-in `fetch`; needs Node ≥ 18):

```bash
npm run build     # node build/harvest.mjs   -> catalog.json + assets/catalog.js (links UNVERIFIED)
npm run verify    # node build/verify_links.mjs -> annotates status; exits non-zero on any BROKEN
npm run analytics # injects the shared Google Analytics include into all HTML pages
npm run all       # build + verify
```

To **view**, just open `index.html` (the catalog is embedded in `assets/catalog.js`,
so no server is needed). `npm run serve` starts one if you prefer.

Google Analytics lives in `assets/analytics.js`. When adding a new standalone
HTML page, run `npm run analytics`; `npm run build` and `npm run all` do this
automatically.

## Updating content

All content lives in `build/sources.json`; the site loads the compiled
`assets/catalog.js`. Two ways to edit — see **[`docs/UPDATING.md`](docs/UPDATING.md)**
for the full guide and section schemas:

- **Studio (GUI):** `npm run studio` → http://localhost:8780/studio → edit
  About / News / People / Papers / Tools → **Save** (compiles locally) →
  **Publish** (commits + pushes). A local-only editor; nothing is hosted.
- **By hand:** edit `build/sources.json`, recompile (`npm run all`, or for
  curated-only edits `regenCurated()` — no network), commit `catalog.json` +
  `assets/catalog.js`, push.

There's also a Claude Code skill at
`.claude/skills/update-dynalytics-site/` so an AI agent can do the above.

## Deploying — dynalytics.lamethods.org

The site is fully static (served from the repo root). To publish on GitHub Pages
under the lamethods org with the custom subdomain:

1. Create the repo and push (e.g. `lamethods/dynalytics`).
2. **Settings → Pages** → Source: *Deploy from a branch*, branch `main`, folder `/`.
3. The committed `CNAME` file sets the custom domain to `dynalytics.lamethods.org`.
4. **DNS** (lamethods.org zone): add a `CNAME` record
   `dynalytics` → `lamethods.github.io`.
5. Enable *Enforce HTTPS* once the certificate is issued.

Commit `catalog.json` + `assets/catalog.js` so the page works immediately and the
weekly `linkcheck.yml` has something to re-verify.

Requires R (≥ 4.1) with `jsonlite`, `xml2`, `httr2`.

### Why the browser User-Agent matters

`saqr.me` (a Hugo/Wowchemy site) returns **HTTP 403** to default library/bot
agents. `verify_links.R` sends a Chrome User-Agent so those links verify
correctly instead of showing up as false breakages — this is essential, not
cosmetic.

## Deploying (decide later)

The site is fully static, so any of these work without changes:

- **GitHub Pages, default domain** — push to a repo, enable Pages → served at
  `https://<user>.github.io/Dynasite/`.
- **saqr.me subpath / subdomain** — same, plus a `CNAME` file (e.g.
  `hub.saqr.me`) and a DNS record.

Either way, commit `catalog.json` so the page works immediately and the weekly
`linkcheck.yml` has something to re-verify.
